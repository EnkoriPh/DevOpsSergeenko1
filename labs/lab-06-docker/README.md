## Теоретические сведения

Контейнеры позволяют запускать приложения в изолированной среде без развёртывания отдельной гостевой операционной системы. В отличие от виртуальных машин, контейнеры используют ядро хостовой системы, поэтому потребляют меньше ресурсов, быстрее запускаются и удобнее переносятся между средами.

Docker предоставляет стандартный формат образов, механизм упаковки зависимостей и удобный интерфейс для управления контейнерами. Изоляция процессов достигается за счёт встроенных возможностей Linux: `namespaces`, `cgroups`, `chroot` и `capabilities`. Благодаря этому контейнеры хорошо подходят для доставки приложений, тестирования и воспроизводимого запуска сервисов.

## Ход выполнения

### 1. Краткие ответы по теме Docker

1. Контейнер отличается от виртуальной машины тем, что использует общее ядро хостовой системы, тогда как виртуальная машина запускает полноценную гостевую ОС.
2. По потреблению ресурсов контейнеры эффективнее: им не требуется гипервизор и отдельная ОС, поэтому накладные расходы ниже.
3. Основные преимущества контейнеров: единый формат поставки ПО, переносимость между средами, удобная работа с зависимостями, изоляция процессов и более экономное использование ресурсов.
4. Docker реализует изоляцию процессов не как отдельный гипервизор, а как удобную оболочку над штатными средствами Linux: `namespaces`, `cgroups`, `chroot` и `capabilities`.
5. В практической работе Docker полезен тем, что даёт единый формат образов, API и CLI для управления контейнерами, а также упрощает воспроизводимый запуск приложений.
6. Если внутри контейнера создать файл через `docker exec -it`, а затем остановить и снова запустить тот же контейнер, файл сохранится в writable-layer контейнера, пока контейнер не удалён.

### 2. Практическая часть 1. Ограничение ресурсов через `systemd-run`

На первом этапе была проверена доступность `systemd`, после чего установлена утилита `stress` для имитации нагрузки на процессор и память.

```bash
systemctl --version
sudo apt update && sudo apt install -y stress
stress --version
```

![Рисунок 1](./assets/figure-01.png)

*Рисунок 1. Проверка `systemd` и установка утилиты `stress`.*

Далее был запущен временный unit `stress-limit.service` через `systemd-run` с ограничениями `CPUQuota=50%` и `MemoryMax=100M`.

```bash
sudo systemd-run --unit=stress-limit --slice=stress.slice \
  -p CPUQuota=50% -p MemoryMax=100M --timeout=300s \
  /usr/bin/stress --cpu 2 --vm 1 --vm-bytes 80M --vm-keep

systemctl status stress-limit.service --no-pager
```

![Рисунок 2](./assets/figure-02.png)

*Рисунок 2. Запуск `stress-limit.service` с ограничением CPU и памяти.*

После запуска были найдены дочерние процессы и проверено текущее потребление ресурсов через `top`.

```bash
pgrep -P 3184 -a
top -b -n 1 -p 3184,3185,3186,3187
```

![Рисунок 3](./assets/figure-03.png)

*Рисунок 3. Просмотр дочерних процессов `stress` и проверка потребления ресурсов.*

Для демонстрации реакции системы на превышение лимита памяти был создан ещё один transient-unit с выделением `1G` памяти при ограничении `100M`. В результате `systemd` завершил процесс с причиной `oom-kill`.

```bash
sudo systemd-run --unit=stress-oom --slice=stress.slice \
  -p CPUQuota=50% -p MemoryMax=100M --timeout=300s \
  /usr/bin/stress --vm 1 --vm-bytes 1G --vm-keep

sleep 5
systemctl status stress-oom.service --no-pager
```

![Рисунок 4](./assets/figure-04.png)

*Рисунок 4. Завершение `stress-oom.service` со статусом `failed` из-за `oom-kill`.*
